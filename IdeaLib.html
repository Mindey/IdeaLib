IdeaLib Documentation

IdeaLib defines a language (IDL -- idea definition language) for quickly describing a business idea (=plan), and quickly getting the evaluation of the its value over time and comparing it with the value over time of other ideas, including the uncertainties and variation in the expected values of investables and deliverables, allowing the modelling of scenarios, and

For that reason, it defines an "Idea" and "IdeaList" object classes in Python. The IdeaList is a sublcass of Python list, and the Idea is a class of its own.

I created the IdeaLib in order to quickly estimate the value of ideas, based on the plan that idea implies. The basic elements of this language are pairs of lines (so the number of lines has to be even), with a prefix made of letters followed by a blank space, denoting some name. For example:

name do1 100
name profit 1000
name do2 10
name profit 10000, waste 100

By this list we mean an I/O process related to our idea:

we plan to start with initial input "do1 100",
and we expect the outcome of "profit 1000",
then we plan that by adding of "do2 10" of input,
we will get the followup output of "profit 10000, waste 100".

The odd lines mean inputs, the even lines mean outputs.

The language interprets the first space in a line as the start of the definition of the input or output. Therefore, we can append, for our convenience, any word to denote the line, for example, we can use symbols 'i:' and 'o:' to denote which is input and which is output, and we don't have to use the same words all the time. We can use our native languae words instead.

i: do1 100
o: profit 1000
in: do2 10
out: profit 10000, waste 100

It is recommended to add a colon after the name, so that it would appear more visually distinctive, as the above.

The comma is used to separate investables and deliverables. For example,

out: profit 10000, waste 100

The space after comma is not necessary, preferred to improve signal-to-noise ratio for a human reader.

Hello World of IDL

For the most simple example, i.e., the hello world, of the IDL, it would be:

: lemon 100
: lemon juice in liters 10

This means an idea of adding 100 lemons and getting 10 liters of their juice. Actually, we can instantiate it, with Idea, and define it:

{python} from IdeaLib import Idea idea = Idea(r''' : lemon 100 : lemon juice in liters 10 ''')

Each pair of lines define a step (a milestone) in the idea relalization process. Internally, each such idea object is represented as a list of tuples, where first element in the tuple means the input, and the second means output. Therefore, the idea above is represented as a single-element list:

{python} [({'lemon': ['100']}, {'lemon juice in liters': ['10']})]

Some interesting things (such as progression over time, and time plotting) can only be done when we have more than one step. For example.

Multi-Step Hello World of IDL

{python} from IdeaLib import Idea i = Idea(r''' i: do1 100 o: profit 1000 in: do2 10 out: profit 10000,waste 100 ''')

This result is encoded in a list of two elements: ```{python} >>> print i

[({'do1': ['100']}, {'profit': ['1000']}), ({'do2': ['10']}, {'profit': ['10000'], 'waste': ['100']})] ```

Note: We can use arbitrary human language, not just English.

Main Methods Available in Idea Object

The main two methods available in Idea object, are .to_df() and .plot():

  • .to_df() converts the idea defined in IDL to a DataFrame.
  • .plot() converts the idea defined in IDL and plots its value.

Note: A single description of idea may have multiple different scenarios encoded.

Defining Multiple Scenarios in IDL

Sometimes, we may want to define several values of investable or deliverable. For example, we may have the lower and upper estimates of our cots for a particular task. In that case, we can use the backslash (\) to denote such different possible values. For example:

i: do1 100\200 o: profit 1000\1500 i: do2 10\15 o: profit 10000\12000, waste 100\500\1000 This means that each of our variables ('do1', 'do2', 'profit', 'waste') may acquire possible scenario values. This is encoded as multiple values of the dictionary encoding it. For instance, the above, is internally translated as:

{python} [({'do1': ['100', '200']}, {'profit': ['1000', '1500']}), ({'do2': ['10', '15']}, {'profit': ['10000', '12000'], 'waste': ['100', '500', '1000']})]

Support of Probability Distributions over Scenarios

COMING SOON... Note: The reason why we switched to using backslash (\) to separate scenarios, is because we will want to use slash (/) for division, in defining distributions, and using a semicolon (;) looks a bit odd, because we use colon (,) to separate different types of resources -- a higher level substance in hierarchy, and thefore using semicolon would be confusing for a reader. Unfortunately, this requires us to enter all the text-based definitions as raw strings, that is, prefix strings with r'...'.

Discrete Distributions

Note1: in the future, it will be possible to denote the probabilities (or ranges of probabilities) associated with a scenario by a sign, e.g., colon (:) sign. A Bernoulli distribution of probabilities across scenarios could be denoted as 50:.1200:.88

Continuous Distributions

Note2: in the future, there will be possible to denote ranges of values by some sign, e.g., tilde (~). For example 1000~2000. In that case, we'd not be able to define other scenarios and only have a probability distribution associated with the range. For instance, 1000~2000:1/(b-a), where variables a,b are the ends of the interval.

Extra Notes on Distributions within Scenarios

It should be possible to have multiple scenarios, and a probability distribution inside each scenario. For example: apples 50~100:.12\100~150:.66\200~250:.22 would mean we have ranges of values, in each case, associated with a probability, that is, with uniform unnormalized distribution within the scenario.

It should also be possible to have formulas for distribution within each scenario. For example, after a vertical bar sign (|):

apples 50~100|p(x)=1/(b-a):.12\100~150|p(x)=1/(b-a):.66\200~250|p(x)=1\(b-a):.22

Support of Value Definitions for Investables/Deliverables

It will be possible to define the values for the items at each step. We may use the at (@) sign for that. For example:

i: apples 500\600 @ apples 1\2
o: oranges 1, apples 1 @ oranges 1, apples 1

The @ sign would split the line in two sides. The left side defines quantities, the right side defines their market price. The probability distributions associated with the market price would have the same syntax for reading, as the ones for the quantities on the left side.

The values will, by default, be inferred to the past to the future.

Weight Parameters iw and ow for Inputs and Outputs

At this point, however, it is only possible to define a constant value for each of the investables and deliverables, by iw and ow paramters in the constructor of an Idea object, for example:

{python} i = Idea(r''' i: do1 100\200 o: profit 1000\1500 i: do2 10\15 o: profit 10000\12000, waste 100\500\1000 ''',iw=[1,4], ow=[2, 1])

In this case, this means, that the inputs do1, do2 have weights 1, 4, and similarly 10, 1 for outputs. However, before adding iw and ow paramters, it is better to look at what kind of data structure (a multi-indexed DataFrame) is generated from the idea description, by .to_df() method.

Generating DataFrames from Idea

One of the core goals of the lib, is to make the generation of the DataFrames for estimation of the value-flow easy. Having an idea defined in IDL allows for automatic generation of Pandas DataFrames, defining the value over time.

The .to_df() method

To generate the idea's DataFrame, use it, for example, i.to_df()

profit waste ivalue ovalue value
do1 do2
150 0.0 1250 NaN 150 2500.000000 2350.000000
0 12.5 11000 533.333333 50 22533.333333 22483.333333

This DataFrame has two columns as input (do1, do2), and two columns as output (profit, waste). There are extra three columns -- hte ivalue, ovalue and value. value = ivalue-ovalue, ivalue = input * iw, ovalue = output * ow.

There are totally two records (two steps). In the first step, we had 150 of "do1" as input, which resulted in 150 of ivalue, because we value "do1" by "1". The ovalue is 2500, because we value the "profit" by "2". Thus, balance is the value. The second row has "do2" as "12.5", but we value it by "4", so, the ivalue is 50.

You may think, why it is 12.5.

Scenario generation in .to_df(scenario=['normal', 'best', 'worst', #])

If we use the backslash symbol (\) to add more than one possible number of a particular item, we can generate more one scenario. By default, .to_df() takes the normal scenario, which is actually the arithmetic mean of the values separated by backslash (\). For example, in the idea above, do2 had the value of 12.5, because it is the arithmetic mean of 10 and 15:

i: do2 10\15
The "best" scenario, is the scenario with minimal values in the input, and maximum values in the output. i.to_df(scenario="best") gives:
profit waste ivalue ovalue value
do1 do2
100 0 1500 NaN 100 3000 2900
0 10 12000 500 40 24500 24460
The "worst" scenario is the scenario with maximal values in the input, and minimal values in the output. i.to_df(scenario="worst") gives:
profit waste ivalue ovalue value
do1 do2
200 0 1000 NaN 200 2000 1800
0 15 10000 100 60 20100 20040

The n-th scenario of the outcomes is generated by providing the number. If all outcome items have at least two possible values, just pass the scenario number, counting from 0:

i.to_df(scenario=0)

The reserved words -- time

time is a reserved word for resource to be used. Once the time parameter is present at least in one of the steps, it becomes possible to estimate the idea's value over time.

The default unit of time is a day. Consider the below. i: do1 100\200, time 10\16 o: profit 1000\1500 i: do2 10\15, time 5\8 o: profit 10000\12000, waste 100\500\1000 This means we will use up 10 to 16 days in the first step, and 5 to 8 days in the second. Let's generate the best scenario:

i.to_df(scenario="best")

profit waste ivalue ovalue value
do1 do2 time
100 0 10 1500 NaN 100 3000 2900
0 10 5 12000 500 40 24500 24460

"dates" parameter

The "dates" parameter allows to add the dates to the DataFrame.

i.to_df(scenario="best", dates=True)

Generating Plots from Idea

Once an idea has the reserved time resource at least once, it becomes plot-able. Let's plot the several scenarios.

Plotting a single Idea Default Scenario Over Time.

{python} from IdeaLib import Idea i = Idea(r''' i: do1 100\200, time 10\16 o: profit 1000\1500 i: do2 10\15, time 5\8 o: profit 10000\12000, waste 100\500\1000 ''',iw=[1,4], ow=[2, 1]) i.plot()

Note that the starting time, while there was no value, yet resources were consumed, is not included. So, the project start time (2015,1,1) may not appear in the plot.

IdeaList Class to Compare Multiple Ideas

The IdeaList object is just like a Python list, but it has a plot method, which allows to compare multiple ideas in it. For example, let's define two ideas. Make sure they all have time resource defined at least in one of the steps.

idea 1 i: do1 100\200, time 10\16 o: profit 1000\1500 i: do2 10\15, time 5\8 o: profit 10000\12000, waste 100\500\1000

idea 2 i: do1 100\200, time 4\8 o: profit 1000\1500 i: do2 10\15, time 9\15 o: profit 10000\12000 i: do1 10\15, time 4\5 o: profit 10\100

Now we can create idea objects and add them to a list.

```{python} from IdeaLib import Idea, IdeaList

i1 = Idea(r''' i: do1 10000, time 106 o: profit 1000500 i: do2 105, time 5 o: profit 100002000, waste 10000000 ''',iw=[1,4], ow=[2, 1])

i2 = Idea(r''' i: do1 10000, time 4 o: profit 1000500 i: do2 105, time 95 o: profit 100002000 i: do1 105, time 4 o: profit 1000 ''',iw=[1,4], ow=[2])

import datetime i1.start_time = datetime.datetime(2015,1,1,0,0) i2.start_time = i1.start_time

ideas = IdeaList([i1, i2]) ```
